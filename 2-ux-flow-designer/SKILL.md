# 2단계: UX 흐름 설계

문제 정의를 바탕으로 사용자가 제품을 어떻게 사용하는지 구조화한다. **흐름과 구조**에 집중하되, 각 화면의 **UI 구조(레이아웃, 컴포넌트 배치, 상태별 화면)**까지 설계하여 프로토타입 단계에서 바로 구현할 수 있도록 한다.

## 사전 확인

- `problem-definition.md`가 있으면 → 읽고 설계 유형 파악 후 시작
- 없으면 → `/1-ux-problem-definer` 권유
- "문제는 이미 정의됐어"라고 하면 → 그 내용 바탕으로 진행

## 설계 유형별 접근

**새 서비스**: 처음부터 설계 (이상적 흐름 → MVP 흐름 → 확장 흐름, 온보딩 별도)
**기존 서비스 개선**: 현재 흐름(AS-IS) 정리 → 이탈·불편 기반 개선 흐름(TO-BE) → 변경 포인트 명시

## 생성하는 6가지 산출물

### 1. User Flow (사용자 흐름도)

목표 달성까지의 단계별 흐름을 텍스트로 구조화. 각 단계마다:
- 사용자 행동
- 시스템이 보여주는 것
- 다음 단계 조건 (분기 포함)

새 서비스: 핵심 흐름 + 보조 흐름(온보딩, 설정)
기존 서비스 개선: AS-IS vs TO-BE 비교

### 2. Edge Cases (예외 케이스)

정상 흐름에서 벗어나는 상황 도출:
- 네트워크 끊김/느린 로딩
- 잘못된 입력, 빈 상태
- 권한 없음, 세션 만료
- 극단적 데이터
- 디바이스/브라우저별 차이
- 최초 vs 재방문 사용자 차이

각 케이스: 발생 상황 / 영향 / 대응 방안

### 3. Information Architecture (정보 구조)

화면 간 관계와 정보의 계층 구조를 트리 구조로 정리.
새 서비스는 전체 IA, 기존 서비스 개선은 변경 부분 중심.

### 4. Screen UI Structure (화면별 UI 구조)

IA에서 정의된 주요 화면의 내부 구조를 와이어프레임 수준으로 설계. 비주얼이 아닌 **레이아웃과 컴포넌트 배치**에 집중. 핵심 흐름에 포함된 화면만 우선.

각 화면마다:

**a) 레이아웃 구조**: Header/Main/Footer 영역, 고정/스크롤 여부

**b) 컴포넌트 목록**: 필요한 UI 요소 (카드, 리스트, 탭, 모달, 바텀시트 등), 역할과 포함 데이터, 배치 순서와 그룹핑

**c) 콘텐츠 우선순위** (시각적 위계)
- 1순위: 가장 먼저 봐야 할 것
- 2순위: 핵심 행동을 지원하는 정보
- 3순위: 부가 정보, 설정, 링크

**d) 인터랙션 패턴**: 주요 터치/클릭 동작과 결과, 화면 전환 방식(push/modal/바텀시트), 입력 방식

**e) 상태별 화면**: 데이터 있음 / 빈 상태(CTA 포함) / 로딩(스켈레톤·스피너) / 에러(재시도 방법)

새 서비스는 주요 화면 모두, 기존 서비스 개선은 변경되는 화면만 AS-IS/TO-BE 비교.

### 5. Global Components & UX Writing (공통 컴포넌트 & UX 라이팅)

**a) 공통 컴포넌트** (화면마다 반복되는 요소)
- 네비게이션 (상단바, 하단 탭바, 사이드 메뉴)
- 공통 피드백 (토스트, 스낵바, 알럿)
- 공통 입력 (검색바, 필터)
- 공통 상태 (로딩 스피너, 에러 화면, 빈 상태 템플릿)

각 공통 컴포넌트의 동작 규칙 명시 (예: "토스트: 하단에서 올라옴, 3초 후 자동 사라짐")

**b) UX 라이팅 가이드**
- 어조: 격식/반말, 친근함 정도
- CTA 버튼 문구 규칙: 동사형 vs 명사형
- 빈 상태 메시지: 상황 설명 + CTA
- 에러 메시지: 원인 + 해결 방법

### 6. Test Cases (테스트 케이스)

설계된 흐름 검증 시나리오. 테이블 형식: ID / 시나리오 / 사전 조건 / 행동 / 기대 결과.

최소 포함: 정상 흐름 완료, 빈 상태 진입, 최초 사용자 온보딩.

## 진행 방법

1. 문제 정의서 기반 설계 유형 확인
2. 핵심 흐름 먼저 설계 (새 서비스: 처음부터 / 개선: AS-IS → TO-BE)
3. 예외 케이스 도출
4. 정보 구조 정리
5. 주요 화면의 UI 구조 설계
6. 공통 컴포넌트·UX 라이팅 정리
7. 테스트 케이스 작성
8. 중간중간 사용자 확인
