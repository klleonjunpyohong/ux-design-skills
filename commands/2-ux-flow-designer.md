# 2단계: UX 흐름 설계

문제 정의를 바탕으로 사용자가 제품을 실제로 어떻게 사용하는지 구조화하는 단계다. **흐름과 구조**에 집중하되, 각 화면의 **UI 구조(레이아웃, 컴포넌트 배치, 상태별 화면)**까지 설계하여 프로토타입 단계에서 바로 구현할 수 있도록 한다.

## 사전 확인

시작하기 전에 `problem-definition.md`가 있는지 확인한다.
- 있으면 → 읽고 설계 유형(새 서비스/개선)을 파악한 뒤 설계 시작
- 없으면 → 사용자에게 문제 정의가 먼저 필요하다고 안내하고, `/1-ux-problem-definer`를 권유한다
- 사용자가 "문제는 이미 정의됐어"라고 구두로 설명하면 → 그 내용을 바탕으로 진행

## 설계 유형별 접근

### 새 서비스 설계일 때

기존 흐름이 없으므로 **처음부터 설계**한다:
- 사용자의 핵심 목표를 달성하는 이상적인 흐름을 먼저 그린다
- MVP에 필요한 최소 흐름과 향후 확장 흐름을 구분한다
- 경쟁 서비스의 흐름을 참고하되, 문제 정의서의 차별점을 반영한다
- 온보딩(최초 사용) 흐름을 별도로 설계한다

### 기존 서비스 개선일 때

현재 흐름을 기준으로 **개선안을 설계**한다:
- 현재 흐름(AS-IS)을 먼저 정리한다
- 이탈 지점/불편 사항을 기반으로 개선 흐름(TO-BE)을 설계한다
- AS-IS → TO-BE 변경 포인트를 명시한다

## 생성하는 6가지 산출물

### 1. User Flow (사용자 흐름도)

사용자의 핵심 목표 달성까지의 단계별 흐름을 텍스트 기반으로 구조화한다.

```
[시작] → [단계1: OO 선택] → [단계2: OO 입력] → [분기: 조건 확인]
                                                    ├─ 성공 → [단계3: 완료]
                                                    └─ 실패 → [에러 처리]
```

각 단계마다:
- 사용자가 하는 행동
- 시스템이 보여주는 것
- 다음 단계로 넘어가는 조건

새 서비스라면 핵심 흐름 + 보조 흐름(온보딩, 설정 등)을 설계한다.
기존 서비스 개선이라면 AS-IS와 TO-BE를 비교하여 보여준다.

### 2. Edge Cases (예외 케이스)

정상 흐름에서 벗어나는 상황들을 도출한다:
- 네트워크 끊김, 느린 로딩
- 잘못된 입력, 빈 상태
- 권한 없음, 세션 만료
- 극단적 데이터 (너무 많거나, 없거나)
- 디바이스/브라우저별 차이
- 최초 사용자 vs 재방문 사용자 차이

각 예외 케이스에 대해:
- 발생 상황
- 사용자에게 미치는 영향
- 권장 대응 방안

### 3. Information Architecture (정보 구조)

화면 간의 관계와 정보의 계층 구조를 정리한다:

```
홈
├── 카테고리
│   ├── 상품 목록
│   │   └── 상품 상세
│   └── 필터/정렬
├── 검색
│   └── 검색 결과
├── 마이페이지
│   ├── 주문 내역
│   └── 설정
└── 장바구니
    └── 결제
```

새 서비스라면 전체 IA를 설계한다.
기존 서비스라면 변경되는 부분을 중심으로 IA를 업데이트한다.

### 4. Screen UI Structure (화면별 UI 구조)

IA에서 정의된 각 화면의 내부 구조를 와이어프레임 수준으로 설계한다. 비주얼 디자인이 아니라 **레이아웃과 컴포넌트 배치**에 집중한다.

모든 화면을 다 설계하지 않는다. **핵심 흐름에 포함된 주요 화면**만 우선 설계하고, 나머지는 필요시 추가한다.

각 화면마다 다음을 정의한다:

**a) 레이아웃 구조**
```
┌─────────────────────┐
│      Header         │  ← 고정/스크롤 여부
├─────────────────────┤
│                     │
│   Main Content      │  ← 스크롤 영역
│                     │
├─────────────────────┤
│   Bottom Nav        │  ← 고정 하단
└─────────────────────┘
```

**b) 컴포넌트 목록**
해당 화면에 필요한 UI 요소를 나열한다:
- 어떤 컴포넌트가 필요한지 (카드, 리스트, 탭, 모달, 바텀시트 등)
- 각 컴포넌트의 역할과 포함 데이터
- 컴포넌트 간 배치 순서와 그룹핑

**c) 콘텐츠 우선순위**
화면 내 정보의 시각적 위계를 정한다:
- 1순위: 사용자가 이 화면에서 가장 먼저 봐야 할 것
- 2순위: 핵심 행동을 지원하는 정보
- 3순위: 부가 정보, 설정, 링크 등

**d) 인터랙션 패턴**
- 주요 터치/클릭 동작과 결과 (탭하면 어디로, 스와이프하면 뭐가)
- 화면 전환 방식 (push, modal, 바텀시트 등)
- 입력 방식 (키보드, 피커, 토글 등)

**e) 상태별 화면**
각 화면의 주요 상태를 정의한다:
- **데이터 있음**: 정상 상태의 화면 구성
- **빈 상태**: 데이터가 없을 때 보여줄 내용과 CTA
- **로딩**: 로딩 중 표시 방식 (스켈레톤, 스피너 등)
- **에러**: 에러 발생 시 안내와 재시도 방법

예시 (마이페이지):
```
## 마이페이지

레이아웃: 스크롤 페이지, 하단 네비게이션 고정

콘텐츠 우선순위:
1. 프로필 요약 (이름, 사진, 등급)
2. 주요 메뉴 그룹 (주문/결제, 혜택, 활동)
3. 설정/고객센터

컴포넌트:
- 프로필 카드: 상단 고정, 사진+이름+등급 배지
- 퀵 액션 바: 포인트, 쿠폰, 찜 (아이콘+숫자)
- 메뉴 그룹 A [주문/결제]: 리스트 아이템 3개
- 메뉴 그룹 B [혜택]: 리스트 아이템 2개
- 메뉴 그룹 C [설정]: 리스트 아이템 4개
- 각 그룹 사이 8px 구분선

인터랙션:
- 프로필 카드 탭 → 프로필 편집 (push)
- 퀵 액션 탭 → 각 상세 페이지 (push)
- 메뉴 아이템 탭 → 해당 페이지 (push)

상태:
- 로그인 전: 로그인 유도 카드 + 제한된 메뉴
- 로그인 후: 전체 프로필 + 풀 메뉴
```

새 서비스라면 주요 화면을 모두 설계한다.
기존 서비스 개선이라면 변경되는 화면만 AS-IS/TO-BE로 비교하여 설계한다.

### 5. Global Components & UX Writing (공통 컴포넌트 & UX 라이팅)

화면 간 공유하는 공통 요소와 텍스트 톤앤매너를 정의한다.

**a) 공통 컴포넌트**
화면마다 반복되는 UI 요소를 한 곳에 정리한다:
- 네비게이션 (상단바, 하단 탭바, 사이드 메뉴)
- 공통 피드백 (토스트, 스낵바, 알럿)
- 공통 입력 (검색바, 필터)
- 공통 상태 (로딩 스피너, 에러 화면, 빈 상태 템플릿)

각 공통 컴포넌트의 동작 규칙을 명시한다:
```
하단 탭바: 5개 탭 (홈, 검색, 주문, 알림, 마이), 현재 탭 강조
토스트: 하단에서 올라옴, 3초 후 자동 사라짐, 탭하면 즉시 닫힘
```

**b) UX 라이팅 가이드**
서비스 전반의 텍스트 톤앤매너를 정한다:
- **어조**: 격식/반말, 친근함 정도 (예: "완료됐어요" vs "완료되었습니다")
- **CTA 버튼 문구 규칙**: 동사형 ("시작하기", "저장하기") vs 명사형 ("다음", "완료")
- **빈 상태 메시지**: 상황 설명 + CTA (예: "아직 주문 내역이 없어요. 첫 주문을 해보세요!")
- **에러 메시지**: 원인 + 해결 방법 (예: "네트워크 연결을 확인해주세요. 다시 시도하려면 아래 버튼을 눌러주세요.")

### 6. Test Cases (테스트 케이스)

설계된 흐름이 제대로 작동하는지 검증할 시나리오:

| ID | 시나리오 | 사전 조건 | 행동 | 기대 결과 |
|----|---------|----------|------|----------|
| TC-01 | 정상 흐름 완료 | 로그인 상태 | 전체 흐름 수행 | 목표 달성 |
| TC-02 | 빈 상태 진입 | 데이터 없음 | 목록 접근 | 안내 메시지 표시 |
| TC-03 | 최초 사용자 온보딩 | 신규 가입 | 앱 첫 실행 | 온보딩 완료 |

## 진행 방법

1. 문제 정의서를 기반으로 설계 유형을 확인한다
2. 핵심 흐름을 먼저 설계한다 (새 서비스면 처음부터, 개선이면 AS-IS → TO-BE)
3. 핵심 흐름이 잡히면 예외 케이스를 도출한다
4. 정보 구조를 정리한다
5. IA 기반으로 주요 화면의 UI 구조를 설계한다
6. 공통 컴포넌트와 UX 라이팅 가이드를 정리한다
7. 테스트 케이스를 작성한다
8. 사용자에게 중간중간 확인을 받으며 진행한다

## 결과물

설계가 완료되면 `ux-flow.md` 파일을 작성한다. 6가지 산출물을 모두 포함한다.
