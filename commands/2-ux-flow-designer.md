# 2단계: UX 흐름 설계

문제 정의를 바탕으로 사용자 흐름과 화면 구조를 설계한다.

## 사전 확인

- `problem-definition.md`가 있으면 → 읽고 설계 유형 파악 후 시작
- 없으면 → `/1-ux-problem-definer` 권유
- "문제는 이미 정의됐어"라고 하면 → 그 내용 바탕으로 진행

## 설계 유형별 접근

**새 서비스**: 처음부터 설계 (이상적 흐름 → MVP 흐름 → 확장, 온보딩 별도)
**기존 서비스 개선**: AS-IS → 이탈·불편 기반 TO-BE → 변경 포인트 명시

## 산출물 폴더 구조 (★ 토큰 절약)

`ux-flow/` 폴더에 6개 파일로 분리 저장. **각 파일을 독립적으로 작성하고 다른 파일은 다시 read하지 않는다.**

```
ux-flow/
├── 01-user-flow.md          ← 사용자 흐름도
├── 02-edge-cases.md         ← 예외 케이스
├── 03-information-architecture.md  ← IA
├── 04-screen-ui-structure.md       ← 화면별 UI 구조
├── 05-global-components.md  ← 공통 컴포넌트 + UX 라이팅
└── 06-test-cases.md         ← 테스트 케이스
```

## 작성 원칙

1. 각 산출물은 **새 파일로 Write** (이미 있으면 append만)
2. 다른 산출물 파일을 **다시 read하지 않는다**
3. 사용자에게 중간 확인을 받으며 진행
4. 한 파일 완료 시 사용자에게 보여주고 피드백 → 다음 파일

---

## 6가지 산출물 (각각 별도 파일)

### 01-user-flow.md — User Flow

목표 달성까지의 단계별 흐름 (텍스트). 각 단계마다:
- 사용자 행동
- 시스템이 보여주는 것
- 다음 단계 조건 (분기 포함)

새 서비스: 핵심 흐름 + 보조 흐름(온보딩, 설정)
개선: AS-IS vs TO-BE 비교

### 02-edge-cases.md — Edge Cases

예외 상황: 네트워크 끊김/느린 로딩, 잘못된 입력, 빈 상태, 권한 없음, 세션 만료, 극단적 데이터, 디바이스/브라우저 차이, 최초 vs 재방문 사용자.

각 케이스: 발생 상황 / 영향 / 대응 방안

### 03-information-architecture.md — Information Architecture

화면 간 관계와 정보의 계층 구조 (트리). 새 서비스: 전체 IA. 개선: 변경 부분 중심.

### 04-screen-ui-structure.md — Screen UI Structure

주요 화면의 와이어프레임 수준 구조. 핵심 흐름에 포함된 화면만.

각 화면마다:
- **a) 레이아웃 구조**: Header/Main/Footer, 고정/스크롤
- **b) 컴포넌트 목록**: 필요한 UI 요소, 역할, 배치 순서
- **c) 콘텐츠 우선순위**: 1순위(핵심 목표) / 2순위(지원) / 3순위(부가)
- **d) 인터랙션 패턴**: 터치/클릭 동작, 화면 전환(push/modal/바텀시트), 입력 방식
- **e) 상태별 화면**: 데이터 있음 / 빈 상태(CTA) / 로딩(스켈레톤·스피너) / 에러(재시도)

### 05-global-components.md — Global Components & UX Writing

**a) 공통 컴포넌트** (화면 반복 요소)
- 네비게이션 (상단바, 하단 탭바, 사이드 메뉴)
- 공통 피드백 (토스트, 스낵바, 알럿)
- 공통 입력 (검색바, 필터)
- 공통 상태 (로딩 스피너, 에러 화면, 빈 상태 템플릿)

각 컴포넌트의 동작 규칙 명시.

**b) UX 라이팅 가이드**
- 어조 (격식/반말, 친근함)
- CTA 문구 규칙 (동사형 vs 명사형)
- 빈 상태 메시지 (상황 + CTA)
- 에러 메시지 (원인 + 해결 방법)

### 06-test-cases.md — Test Cases

설계된 흐름 검증 시나리오 (테이블: ID/시나리오/사전 조건/행동/기대 결과).

최소 포함: 정상 흐름 완료, 빈 상태 진입, 최초 사용자 온보딩.

## 진행 방법

1. `ux-flow/` 폴더 생성
2. `01-user-flow.md`부터 순서대로 작성 (각각 새 파일로 Write)
3. 각 파일 완성 시 사용자에게 보여주고 피드백 → 다음 파일
4. 다른 산출물 파일을 다시 읽지 않는다 (필요 시 사용자에게 물어봄)
