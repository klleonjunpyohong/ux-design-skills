# 1단계: UX 문제 정의

화면을 만들기 전에 "무엇을 왜 만드는지"를 먼저 명확히 하는 단계다. 좋은 UX는 좋은 질문에서 시작한다.

## 설계 유형 판별

사용자의 요청을 듣고 먼저 어떤 상황인지 파악한다:

- **새 서비스 설계**: "~앱을 만들고 싶어", "새로운 서비스를 기획 중이야"
- **기존 서비스 개선**: "전환율이 낮아", "이탈률을 줄이고 싶어", "등록 과정을 개선하고 싶어"

명확하지 않으면 물어본다: "새로 만드는 서비스인가요, 기존 서비스를 개선하려는 건가요?"

## 핵심 질문 프레임워크

### 새 서비스 설계일 때

**1. 비전과 목적**
- 이 서비스를 왜 만들려고 하는가? 어떤 문제를 해결하고 싶은가?
- 이 서비스가 성공하면 사용자의 일상이 어떻게 달라지는가?
- 비슷한 서비스가 이미 있는가? 있다면 왜 새로 만들려 하는가?

**2. 사용자 이해 (페르소나)**
- 핵심 사용자는 누구인가? (연령, 직업, 기술 숙련도)
- 사용자가 이 서비스를 사용하게 되는 맥락(상황, 시간, 장소)은?
- 사용자가 현재 이 문제를 어떻게 해결하고 있는가? (기존 대안)
- 사용자의 핵심 JTBD(Job to be Done)는? ("~할 때, ~하고 싶다, 그래서 ~할 수 있도록")
- 예상 사용 빈도와 패턴은? (매일/주간/이벤트성)

**3. 경쟁 서비스 벤치마킹**
- 참고하거나 경쟁하는 서비스가 있는가? (URL, 앱 이름)
- 있다면 그 서비스의 강점과 약점은?
- 그 서비스와 비교했을 때 우리의 차별점은?
- 참고 서비스의 URL이나 스크린샷이 제공되면 → 화면 구조, 핵심 흐름, UX 강점/약점을 분석한 뒤 요약한다
- 없으면 → 해당 도메인에서 일반적으로 참고할 만한 서비스를 1~2개 제안한다

**4. 핵심 기능**
- 이 서비스에서 가장 중요한 기능 1~3가지는?
- 첫 버전(MVP)에 반드시 들어가야 하는 것과 나중에 추가할 것은?

**5. 제약 조건**
- 기술적 제약이 있는가? (플랫폼, 기존 시스템 연동 등)
- 비즈니스 제약이 있는가? (일정, 예산, 규정 등)

**6. 성공 기준**
- 이 서비스가 성공했다고 판단할 기준은?
- 출시 후 어떤 지표를 추적하고 싶은가?

### 기존 서비스 개선일 때

**1. 사용자 이해**
- 핵심 사용자는 누구인가?
- 사용자가 이 서비스를 사용하는 맥락(상황)은?

**2. 현황 파악**
- 현재 어떤 흐름으로 운영되고 있는가?
- 사용자가 가장 많이 이탈하는 지점은 어디인가? (데이터가 있는가?)
- 사용자 불만이나 피드백에서 반복되는 패턴은?

**3. 개선 목표**
- 가장 시급하게 개선하고 싶은 부분은?
- 어떤 지표를 개선하고 싶은가? (전환율, 이탈률, 완료율 등)

**4. 제약 조건**
- 기술적/비즈니스 제약이 있는가?
- 기존 시스템에서 바꿀 수 없는 부분이 있는가?

## 진행 방법

1. 사용자의 초기 설명을 듣는다
2. 설계 유형(새 서비스/개선)을 판별한다
3. 해당 유형의 질문 중 아직 답이 없는 것들을 **대화형으로** 물어본다
   - 한 번에 모든 질문을 쏟아내지 않는다
   - 사용자의 답변에 따라 후속 질문을 조절한다
   - 3~5회 정도의 대화로 핵심을 파악한다
4. 충분한 정보가 모이면 문제 정의서를 작성한다

## 입력 지원

사용자가 다음을 제공하면 더 정확한 문제 정의가 가능하다:

**새 서비스**: 참고 서비스 URL, 시장 조사 자료, 사업 계획서
**기존 서비스**: 서비스 스크린샷, PRD, 사용자 피드백/리뷰 데이터, 분석 데이터

이런 자료가 있으면 먼저 분석한 뒤 질문한다. 없어도 대화만으로 충분히 진행 가능하다.

## 결과물

대화가 충분히 이루어지면 `problem-definition.md` 파일을 작성한다:

```markdown
# 문제 정의서

## 설계 유형
[새 서비스 설계 / 기존 서비스 개선]

## 제품/기능 개요
[한두 문장으로 요약]

## 핵심 사용자 (페르소나)
- 누구: [연령, 직업, 기술 숙련도]
- 사용 맥락: [어떤 상황/시간/장소에서 쓰는가]
- JTBD: [~할 때, ~하고 싶다, 그래서 ~할 수 있도록]
- 사용 빈도: [매일/주간/이벤트성]
- 핵심 목표: [무엇을 달성하려 하는가]

## 경쟁/참고 서비스 벤치마킹
- 참고 서비스: [서비스명, URL]
- 강점: [잘하고 있는 점]
- 약점: [부족한 점]
- 우리의 차별점: [다르게 할 것]

## 문제 정의
- 핵심 문제: [해결하려는 문제]
### (새 서비스인 경우)
- 사용자의 현재 해결 방법: [기존 대안과 한계]
- 핵심 가치 제안: [이 서비스가 제공하는 차별적 가치]
- MVP 핵심 기능: [첫 버전에 필요한 기능]
### (기존 서비스 개선인 경우)
- 현재 이탈/불편 지점: [구체적 지점들]
- 기존 흐름의 문제: [무엇이 잘 안 되고 있는가]

## 제약 조건
- [기술적/비즈니스 제약 나열]

## 성공 기준
- [측정 가능한 목표 나열]
```
