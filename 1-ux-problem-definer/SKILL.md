# 1단계: UX 문제 정의

화면을 만들기 전에 "무엇을 왜 만드는지"를 먼저 명확히 한다. 좋은 UX는 좋은 질문에서 시작한다.

## 설계 유형 판별

- **새 서비스 설계**: "~앱을 만들고 싶어", "새로운 서비스 기획 중"
- **기존 서비스 개선**: "전환율이 낮아", "이탈률 줄이고 싶어"

명확하지 않으면 물어본다: "새로 만드는 서비스인가요, 기존 서비스를 개선하려는 건가요?"

## 핵심 질문 프레임워크

### 새 서비스 설계일 때

**1. 비전과 목적**
- 왜 만드는가? 어떤 문제를 해결하는가?
- 성공하면 사용자 일상이 어떻게 달라지는가?
- 이미 있는 서비스 대비 왜 새로 만드는가?

**2. 사용자 이해 (페르소나)**
- 누구인가? (연령, 직업, 기술 숙련도)
- 사용 맥락 (상황, 시간, 장소)
- 현재 해결 방법 (기존 대안)
- 핵심 JTBD: "~할 때, ~하고 싶다, 그래서 ~할 수 있도록"
- 사용 빈도 (매일/주간/이벤트성)

**3. 경쟁 서비스 벤치마킹**
- 참고/경쟁 서비스 (URL, 앱 이름)
- 강점/약점, 우리의 차별점
- URL·스크린샷 제공 시 → 화면 구조·핵심 흐름·UX 강점/약점 분석
- 없으면 → 도메인에 맞는 참고 서비스 1~2개 제안

**4. 핵심 기능**
- 가장 중요한 기능 1~3가지
- MVP vs 확장

**5. 제약 조건**
- 기술적 제약 (플랫폼, 연동)
- 비즈니스 제약 (일정, 예산, 규정)

**6. 성공 기준**
- 성공 판단 기준
- 추적할 지표

### 기존 서비스 개선일 때

**1. 사용자 이해**: 핵심 사용자와 사용 맥락
**2. 현황 파악**: 현재 흐름, 이탈 지점, 반복되는 불만
**3. 개선 목표**: 시급한 부분, 개선 지표
**4. 제약 조건**: 기술·비즈니스 제약, 바꿀 수 없는 부분

## 진행 방법

1. 사용자 설명 듣기
2. 설계 유형 판별
3. 해당 유형 질문을 **대화형으로** 물어본다
   - 한 번에 다 쏟아내지 않기
   - 답변에 따라 후속 질문 조절
   - 3~5회 대화로 핵심 파악
4. 충분히 모이면 문제 정의서 작성

## 입력 지원

사용자가 제공하면 분석 후 질문:
- 새 서비스: 참고 서비스 URL, 시장 조사, 사업 계획서
- 기존 서비스: 스크린샷, PRD, 피드백/리뷰, 분석 데이터

없어도 대화만으로 진행 가능.

## 결과물

`problem-definition.md` 파일을 작성한다. 다음 섹션을 포함:

- 설계 유형
- 제품/기능 개요
- 핵심 사용자 (페르소나) — 누구/사용 맥락/JTBD/사용 빈도/핵심 목표
- 경쟁/참고 서비스 벤치마킹 — 참고 서비스/강점/약점/차별점
- 문제 정의 — 핵심 문제/현재 해결 방법 or 이탈 지점/가치 제안 or 흐름 문제/MVP 기능
- 제약 조건
- 성공 기준

## CLAUDE.md 업데이트

문제 정의서 작성과 동시에 프로젝트 루트의 `CLAUDE.md`도 생성/갱신:
- 없으면 신규 생성 (`/ux-design`의 CLAUDE.md 템플릿 참조)
- 진행 상태에서 1단계 ✅로 표시
- 핵심 결정사항에 서비스 성격, 핵심 사용자(한 줄), 핵심 문제 기록
- 사용자 피드백으로 결정이 바뀐 경우 의사결정 로그에 추가
