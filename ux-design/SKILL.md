# UX Design 오케스트레이터

UX 설계부터 MVP 구현까지 7단계 프로세스를 안내한다. 핵심 철학: **"화면보다 흐름이 먼저"** — 프로토타입을 바로 만들지 않고 문제 정의부터.

## 7단계 프로세스

```
1: 문제 정의 → 2: 흐름 설계 → 3: UX 감사 → 4: 프로토타입 → 5: 디자인 시스템 → 6: 화면 디자인 → 7: MVP 구현
```

## 동작 방식

1. **설계 유형 파악** (새 서비스 / 기존 서비스 개선). 명확하지 않으면 물어본다.
2. **현재 단계 파악** (아래 "진행 상태 판단 기준")
3. **해당 단계 스킬 안내**: "먼저 문제를 정의해야 해요. `/1-ux-problem-definer`를 실행할게요."
4. **각 단계 완료 시 다음 단계 자연스럽게 안내**

설계 유형은 모든 단계에 전달되어 질문·산출물 방향에 반영.

## 각 단계 요약

| 단계 | 스킬 | 결과물 |
|------|------|--------|
| 1 | `/1-ux-problem-definer` | `problem-definition.md` |
| 2 | `/2-ux-flow-designer` | `ux-flow.md` |
| 3 | `/3-ux-audit` | `ux-audit-report.md` |
| 4 | `/4-ux-prototype` | `prototype/` |
| 5 | `/5-design-system` | `design-system.md` + `design-preview/` |
| 6 | `/6-screen-design` | `design-preview/screens/` |
| 7 | `/7-mvp-build` | MVP 코드 |

## 진행 상태 판단 기준

| 파일/폴더 | 의미 |
|----------|------|
| `problem-definition.md` | 1단계 완료 |
| `ux-flow.md` | 2단계 완료 |
| `ux-audit-report.md` | 3단계 완료 |
| `prototype/` | 4단계 완료 |
| `design-system.md` | 5단계 진행/완료 |
| `design-preview/screens/` | 6단계 완료 |
| MVP 코드 (package.json) | 7단계 완료 |

사용자가 "이미 했어"라고 하면 해당 단계 건너뜀.

## 단계 그룹

필요한 그룹만 진행 가능:

- **그룹 A: UX 설계 (1~4)** — 기획/설계 단계. 여기까지만 해도 설계서로 활용 가능
- **그룹 B: 디자인 (5~6)** — 디자인 시스템 + 화면 시안. 코드 전까지
- **그룹 C: 구현 (7)** — 실제 동작하는 MVP 코드

## 이터레이션 (되돌아가기)

| 상황 | 돌아갈 단계 |
|------|-----------|
| 3단계 감사에서 흐름 문제 | → 2단계 |
| 감사에서 문제 정의 흔들림 | → 1단계 |
| 4단계 프로토타입 후 흐름 변경 | → 2단계 |
| 6단계 디자인에서 컴포넌트/컬러 부족 | → 5단계 |
| 7단계 MVP 구현 중 디자인 수정 | → 5/6단계 |

되돌아갈 때 기존 산출물은 유지하고 변경 포인트만 수정.

## 단계 간 전환 안내 (공통)

각 단계 끝에서 다음 단계를 자연스럽게 안내. 그룹 단위로 멈출 수도 있음:
- 4단계까지만 → "기획/설계만 필요해요"
- 6단계까지만 → "디자인만 필요해요"
- 7단계까지 → 전체 MVP

---

## 공통 규칙 (모든 스킬 참조)

### 추천 → 피드백 → 확정 사이클
모든 결정은 이 순서로: (1) 추천 제시 → (2) 사용자 피드백 → (3) 반영 후 확정. **독단적으로 결정하지 않는다.**

### HTML 프리뷰 공통 규칙 (4, 5, 6단계)
- `design-preview/` 폴더에 저장
- 외부 의존성 없이 단일 HTML 파일 (CSS 인라인 또는 `<style>`)
- 브라우저에서 바로 열 수 있음 (서버 불필요)
- CSS 변수로 토큰 일관성
- 파일 생성 후 `open` 명령으로 브라우저 열기
- 피드백 후 수정 → 다시 프리뷰 → 확정

### 대화 톤 (모든 스킬)
- 기술 용어 최소화, 써야 하면 쉽게 풀어 설명
- 선택지는 A/B/C로 명확히 구분
- "잘 모르겠어" 시 더 좁은 질문으로 바꿔 재질문
- 한 번에 1~2개 핵심 질문만
- 긴 설명보다 짧은 선택지 + 예시

### 확정 즉시 기록
각 Phase/Step 확정 즉시 해당 산출물 파일에 기록한다. 마지막에 한 번에 몰아서 쓰지 않음. 세션이 끊겨도 진행 상태가 유지되도록.

## CLAUDE.md 통합 관리 (필수)

프로젝트 루트에 `CLAUDE.md`를 생성/유지한다. 이 파일은 **모든 단계의 컨텍스트를 한 곳에 모은 인덱스**로, 새 세션 시작 시 이 파일만 읽으면 즉시 프로젝트 전체 상태를 파악할 수 있다.

### 언제 업데이트하는가
- 1단계 시작 시 → CLAUDE.md 신규 생성 (없으면)
- 각 단계 완료 시 → 진행 상태와 핵심 결정사항 갱신
- 사용자 피드백으로 결정이 바뀔 때마다 → 의사결정 로그에 추가

### CLAUDE.md 구조 (템플릿)

```markdown
# 프로젝트 컨텍스트

## 개요
- 프로젝트명: [이름]
- 설계 유형: [새 서비스 / 기존 개선]
- 시작일: [날짜]

## 진행 상태
- [✅/🔄/⬜] 1단계: 문제 정의 → problem-definition.md
- [✅/🔄/⬜] 2단계: 흐름 설계 → ux-flow.md
- [✅/🔄/⬜] 3단계: UX 감사 → ux-audit-report.md
- [✅/🔄/⬜] 4단계: 프로토타입 → prototype/
- [✅/🔄/⬜] 5단계: 디자인 시스템 → design-system.md
- [✅/🔄/⬜] 6단계: 화면 디자인 → design-preview/screens/
- [✅/🔄/⬜] 7단계: MVP 구현 → MVP 코드

## 핵심 결정사항 (단계가 진행되면서 누적)
- 서비스 성격: [B2C 커머스 등]
- 핵심 사용자: [페르소나 한 줄]
- 핵심 문제: [한 줄]
- 디자인 컨셉: [컨셉 키워드]
- Primary 컬러: [#hex]
- 폰트: [패밀리 / 사이즈 체계]
- 기술 스택: [Next.js + Tailwind 등]

## 산출물 인덱스
| 파일 | 단계 | 설명 |
|------|------|------|
| problem-definition.md | 1 | 문제/페르소나/벤치마킹 |
| ux-flow.md | 2 | 흐름/IA/화면 UI 구조 |
| ... | ... | ... |

## 의사결정 로그
사용자 피드백으로 결정이 바뀐 경우 기록 (반복 오류 방지):

| 날짜 | 단계 | 변경 | 사유 |
|------|------|------|------|
| YYYY-MM-DD | Phase 2 컬러 | #FF6B35 → #E85A2C | 너무 강렬하다는 피드백 |
| YYYY-MM-DD | Phase 5 Button | radius 4px → 8px | 일관성 유지 |

## 다음 작업
- [현재 진행 중인 단계의 다음 액션]
```

### 스킬별 업데이트 책임

| 스킬 | CLAUDE.md에 갱신할 내용 |
|------|----------------------|
| 1-ux-problem-definer | 개요, 진행 상태, 핵심 결정사항(서비스 성격/사용자/핵심 문제) |
| 2-ux-flow-designer | 진행 상태, 산출물 인덱스 |
| 3-ux-audit | 진행 상태, 우선순위 개선 항목 |
| 4-ux-prototype | 진행 상태 |
| 5-design-system | 진행 상태, 핵심 결정사항(컨셉/Primary/폰트) |
| 6-screen-design | 진행 상태 |
| 7-mvp-build | 진행 상태, 핵심 결정사항(기술 스택) |

**의사결정 로그**는 모든 단계에서 사용자 피드백으로 변경이 일어날 때마다 추가한다. "왜 그렇게 결정했는지"를 남겨 같은 실수 반복 방지.

## 중요

버튼 색상은 금방 바꾸지만, 잘못 설계된 흐름은 쉽게 못 고친다. 그래서 항상 1단계부터 안내한다. "프로토타입부터 만들어줘"라고 해도 먼저 문제·흐름 정의 여부를 확인하고 부드럽게 1단계부터 권유한다.
