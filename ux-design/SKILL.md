# UX Design 오케스트레이터

UX 설계부터 MVP 구현까지 7단계 프로세스를 안내한다. 핵심 철학: **"화면보다 흐름이 먼저"** — 프로토타입을 바로 만들지 않고 문제 정의부터.

## 7단계 프로세스

```
1: 문제 정의 → 2: 흐름 설계 → 3: UX 감사 → 4: 프로토타입 → 5: 디자인 시스템 → 6: 화면 디자인 → 7: MVP 구현
```

## 동작 방식

1. **설계 유형 파악** (새 서비스 / 기존 서비스 개선). 명확하지 않으면 물어본다.
2. **현재 단계 파악** (`CLAUDE.md` 또는 산출물 폴더 존재 여부)
3. **해당 단계 스킬 안내**: "먼저 문제를 정의해야 해요. `/1-ux-problem-definer`를 실행할게요."
4. **각 단계 완료 시 다음 단계 자연스럽게 안내**

## 각 단계 요약

| 단계 | 스킬 | 결과물 |
|------|------|--------|
| 1 | `/1-ux-problem-definer` | `problem-definition.md` |
| 2 | `/2-ux-flow-designer` | `ux-flow/` 폴더 (6개 파일) |
| 3 | `/3-ux-audit` | `ux-audit-report.md` |
| 4 | `/4-ux-prototype` | `prototype/` |
| 5 | `/5-design-system` | `design-system/` 폴더 (6개 파일) + `design-preview/` |

## 단계 그룹

- **그룹 A: UX 설계 (1~4)** — 기획/설계
- **그룹 B: 디자인 (5~6)** — 디자인 시스템 + 화면 시안
- **그룹 C: 구현 (7)** — MVP 코드

필요한 그룹만 진행 가능.

## 공통 규칙 (모든 스킬 참조)

### 추천 → 피드백 → 확정 사이클
모든 결정은: (1) 추천 제시 → (2) 사용자 피드백 → (3) 반영 후 확정. **독단적으로 결정하지 않는다.**

### HTML 프리뷰 공통 규칙 (4, 5, 6단계)
- `design-preview/` 폴더에 저장
- 외부 의존성 없이 단일 HTML 파일
- 브라우저에서 바로 열 수 있음
- CSS 변수로 토큰 일관성
- 파일 생성 후 `open` 명령으로 열기

### 대화 톤
- 기술 용어 최소화
- 선택지는 A/B/C로 명확히
- "잘 모르겠어" 시 더 좁은 질문으로
- 한 번에 1~2개 핵심 질문만

### 토큰 절약: 산출물 파일 다루기 (★ 중요)

**원칙: 한 번 쓴 파일은 다시 읽지 않는다.**

각 단계의 산출물은 폴더로 분리되어 있어, 새 단계를 진행할 때 이전 단계 파일을 다시 읽을 필요가 없다. 필요한 정보는 사용자에게 물어보거나, 해당 단계의 새 파일만 작성한다.

**구체적 규칙:**

1. **새 파일 우선**: 새 Phase/Step 결과는 새 파일로 저장 (Write)
2. **기존 파일 read 금지**: 같은 폴더의 다른 파일을 다시 읽지 않는다
3. **꼭 읽어야 할 때**: `offset/limit`으로 필요한 줄만 (예: 헤더 10줄)
4. **전체 read 금지**: 산출물 파일 전체를 그대로 read하지 않는다
5. **append-only**: 기존 파일에 추가할 때는 끝에 새 섹션만 추가 (Edit의 old_string은 마지막 줄로)

**예외:**
- 다음 단계 스킬이 명시적으로 이전 단계의 특정 파일을 입력으로 받는 경우만 read
- 사용자가 "이전 결과 다시 보여줘"라고 명시적으로 요청한 경우

## CLAUDE.md (프로젝트 인덱스)

프로젝트 루트에 `CLAUDE.md`를 만들어 **짧은 인덱스**로 유지한다. 새 세션에서 이 파일만 봐도 어디까지 진행됐는지 알 수 있도록.

### CLAUDE.md 구조 (짧게 유지)

```markdown
# 프로젝트 컨텍스트

## 개요
- 프로젝트명: [이름]
- 설계 유형: [새 서비스 / 기존 개선]

## 진행 상태
- ✅ 1단계: 문제 정의
- ✅ 2단계: 흐름 설계
- 🔄 3단계: UX 감사 (진행 중)
- ⬜ 4~7단계

## 산출물 위치
| 단계 | 위치 |
|------|------|
| 1 | problem-definition.md |
| 2 | ux-flow/ |
| 3 | ux-audit-report.md |
| 4 | prototype/ |
| 5 | design-system/ |
| 6 | design-preview/screens/ |
| 7 | [코드 경로] |

## 다음 작업
- [현재 단계의 다음 액션 한 줄]
```

**원칙:**
- 핵심 결정사항이나 상세 내용은 **CLAUDE.md에 넣지 않는다** (각 산출물 파일에)
- 매 단계 완료 시 진행 상태와 다음 작업만 갱신
- 의사결정 로그는 별도 파일 `decisions.md`에 (변경 사유 기록)

### decisions.md (의사결정 로그, 선택사항)

사용자 피드백으로 결정이 바뀐 경우, 프로젝트 루트 `decisions.md`에 한 줄씩 append:

```markdown
- 2026-04-07 | Phase 2 컬러 | #FF6B35 → #E85A2C | 너무 강렬하다는 피드백
- 2026-04-07 | Phase 5 Button | radius 4px → 8px | 일관성 유지
```

기존 내용 read 없이 끝에 한 줄만 append. 같은 실수 반복 방지가 목적.

## 중요

버튼 색상은 금방 바꾸지만, 잘못 설계된 흐름은 쉽게 못 고친다. 항상 1단계부터 안내한다. "프로토타입부터 만들어줘"라고 해도 먼저 문제·흐름 정의 여부를 확인하고 1단계부터 권유한다.
