# 5단계: 디자인 시스템 → MVP

기획 문서를 기반으로 시각적 디자인 시스템을 구축하고, 실제 동작하는 MVP를 제작하는 단계다.

핵심 원칙: **독단적으로 결정하지 않는다.** 모든 디자인 결정은 추천 → 사용자 확인 → 반영의 흐름으로 진행한다.

## 사전 확인

시작하기 전에 기획 문서를 찾는다:
- 사용자가 직접 지정한 md 파일
- 프로젝트 내 `problem-definition.md`, `ux-flow.md` 등 이전 단계 산출물
- 없으면 사용자에게 기획 문서를 요청한다

기획 문서를 읽고 다음을 파악한다:
- 서비스의 성격 (B2B, B2C, 커머스, 금융, 교육, 소셜 등)
- 핵심 사용자 (연령대, 성향)
- 핵심 기능과 화면 목록
- 분위기/톤 관련 단서

특히 `ux-flow.md`가 있으면 다음을 설계도로 활용한다:
- **화면별 UI 구조** → Phase 5 컴포넌트의 출발점 (처음부터 논의하지 않고 기존 설계 기반으로 보완)
- **공통 컴포넌트** → Phase 5에서 그대로 반영 (네비, 토스트 등 이미 정의된 것)
- **UX 라이팅 가이드** → MVP 코드의 텍스트에 톤앤매너, CTA 문구 규칙 적용

`ux-audit-report.md`가 있으면 감사에서 발견된 이슈를 MVP에 반영한다:
- 접근성 이슈 → 코드에 직접 반영
- UI 구조 개선안 → 컴포넌트/레이아웃에 반영
- 이탈 지점 개선안 → 흐름에 반영

## 진행 순서

총 7단계로 진행하며, 각 단계마다 반드시 **추천 제시 → 사용자 피드백 → 확정** 사이클을 거친다.

```
Phase 1: 디자인 컨셉  →  Phase 2: 컬러  →  Phase 3: 폰트  →  Phase 4: 화면 설정
    →  Phase 5: 컴포넌트  →  Phase 6: 아이콘  →  Phase 7: MVP 제작
```

## Phase 1: 디자인 컨셉

기획 문서를 분석해서 전체적인 디자인 방향을 제안한다.

**추천할 때 포함하는 항목:**
- 전체 분위기 키워드 (예: "깔끔하고 신뢰감 있는", "따뜻하고 친근한", "모던하고 미니멀한")
- 참고할 만한 디자인 스타일 (예: Material Design 느낌, iOS 네이티브 느낌, 글래스모피즘 등)
- 둥근 모서리 vs 각진 모서리, 그림자 사용 여부 같은 시각적 톤
- 2~3가지 방향을 제시해서 사용자가 선택하거나 조합할 수 있게 한다

**레퍼런스 입력 지원:**

사용자가 원하는 분위기를 말로 설명하기 어려울 수 있다. 다음을 안내한다:
- 마음에 드는 앱/웹사이트의 **URL**을 공유해도 된다
- **스크린샷 이미지**를 첨부해도 된다 (파일 경로나 이미지 직접 첨부)
- 여러 개를 섞어서 "이 사이트의 컬러 + 저 앱의 레이아웃" 같은 조합도 가능

레퍼런스가 제공되면:
1. URL → WebFetch로 접근해서 디자인 특성을 분석한다
2. 이미지 → 시각적 특성(컬러톤, 레이아웃, 타이포그래피 스타일)을 분석한다
3. 분석 결과를 바탕으로 더 정확한 컨셉 추천을 한다

레퍼런스가 없어도 괜찮다. 기획서만으로 충분히 추천 가능하다.

**대화 예시:**
> 기획서를 분석해봤습니다. 건강 관리 앱이니 다음 3가지 방향을 제안드려요:
>
> **A. 클린 & 트러스트** — 흰 배경, 넓은 여백, 각진 카드. 병원/의료 느낌.
> **B. 웜 & 프렌들리** — 부드러운 베이지 톤, 둥근 모서리, 일러스트 활용. 일상 건강 앱 느낌.
> **C. 다크 & 프리미엄** — 어두운 배경, 네온 포인트, 피트니스/운동 앱 느낌.
>
> 혹시 참고하고 싶은 앱이나 웹사이트가 있으면 URL이나 스크린샷을 공유해주세요!
> 없으면 위 방향 중에서 골라주셔도 됩니다. 조합도 가능해요.

사용자가 선택하면 구체적인 키워드로 정리해서 확인받는다.

**확정 즉시 → `design-system.md`에 "디자인 컨셉" 섹션을 기록한다.**

---

## Phase 2: 컬러

Phase 1에서 확정된 컨셉을 바탕으로 컬러 팔레트를 제안한다.

**제안 항목:**
- Primary (메인 컬러) — 브랜드 정체성
- Secondary (보조 컬러) — 보완색
- Accent (강조 컬러) — CTA 버튼 등
- Background (배경) — 메인 배경, 카드 배경
- Text (텍스트) — 제목, 본문, 보조 텍스트
- Semantic (의미 컬러) — Success(성공), Warning(경고), Error(에러), Info(정보)
- 다크 모드 지원 여부도 이 단계에서 논의

**확정 후 → HTML 프리뷰 생성**

`design-preview/colors.html` 파일을 만들어 브라우저에서 바로 확인할 수 있게 한다:
- 각 컬러를 네모 박스로 나열 (헥스값, 이름 표시)
- 밝은 배경 위의 텍스트 / 어두운 배경 위의 텍스트 조합 미리보기
- 버튼 컬러 조합 미리보기
- 다크 모드 포함 시 토글 버튼으로 라이트/다크 전환 가능

프리뷰를 보여준 뒤 피드백을 받고, 수정이 필요하면 조정 후 다시 프리뷰한다.

**확정 즉시 → `design-system.md`에 "컬러 팔레트" 섹션을 기록한다.**

---

## Phase 3: 폰트

서비스 성격과 디자인 컨셉에 맞는 폰트를 제안한다.

**제안 항목:**
- 제목(Heading) 폰트
- 본문(Body) 폰트
- 폰트 크기 체계 (h1~h6, body, caption, small)
- 폰트 굵기 (Regular, Medium, Bold 등 사용할 두께)
- 줄간격(line-height) 기준
- 한글/영문 폰트 각각 제안 (Google Fonts 등 무료 웹폰트 기준)

**확정 후 → HTML 프리뷰 생성**

`design-preview/typography.html` 파일을 만든다:
- 각 크기별 실제 텍스트 샘플 (h1~h6, body, caption)
- 한글 + 영문 + 숫자 혼합 샘플
- 굵기별 비교
- 실제 서비스에서 사용될 법한 예시 문구로 표시

프리뷰를 보여준 뒤 피드백을 받는다.

**확정 즉시 → `design-system.md`에 "타이포그래피" 섹션을 기록한다.**

---

## Phase 4: 디바이스 화면 설정 & 스페이싱 시스템

웹과 모바일 양쪽을 고려한 화면 설정과 간격 체계를 제안한다.

**제안 항목:**
- 지원 디바이스 목록 (모바일, 태블릿, 데스크톱)
- 브레이크포인트 (화면 크기 기준점)
- 각 디바이스별 기본 레이아웃 전략 (1단, 2단, 사이드바 등)
- 모바일 우선(mobile-first) vs 데스크톱 우선 방향

**스페이싱 시스템 (간격 체계):**

4px 또는 8px 기반 그리드로 일관된 간격 체계를 정한다:

```
--spacing-2xs: 2px
--spacing-xs:  4px
--spacing-sm:  8px
--spacing-md:  16px
--spacing-lg:  24px
--spacing-xl:  32px
--spacing-2xl: 48px
--spacing-3xl: 64px
```

각 간격의 용도를 명시한다:
- 컴포넌트 내부 패딩 (xs~md)
- 컴포넌트 간 간격 (md~lg)
- 섹션 간 간격 (xl~3xl)
- 화면 좌우 여백 (md~lg)

**접근성 기준:**
- 최소 터치 영역: 44x44px
- 텍스트 색상 대비: 4.5:1 이상 (WCAG AA)
- 포커스 표시: 모든 인터랙티브 요소에 포커스 링

이 단계는 시각적 프리뷰보다 표/목록으로 정리해서 보여주고 확인받는다.

**확정 즉시 → `design-system.md`에 기록한다.**

## Phase 5: 기본 컴포넌트

확정된 컬러, 폰트, 화면 설정을 조합해서 기본 UI 컴포넌트를 만든다.

**이전 산출물 활용:**
`ux-flow.md`에 "공통 컴포넌트"와 "화면별 UI 구조"가 정의되어 있으면:
- 이미 정의된 공통 컴포넌트(네비, 토스트, 검색바 등)를 그대로 목록에 포함한다
- 화면별 UI 구조에서 사용된 컴포넌트(카드, 리스트, 탭 등)를 추출하여 추가한다
- 처음부터 논의하지 않고, 기존 설계를 기반으로 "이 목록으로 가도 될까요?" 형태로 확인받는다

**제작할 컴포넌트 목록:**
기획서의 화면 목록을 분석해서 필요한 컴포넌트를 나열하고, 우선순위를 사용자와 합의한다.

일반적으로 포함되는 기본 컴포넌트:
- 버튼 (Primary, Secondary, Outline, Text, Disabled 상태)
- 입력 필드 (텍스트, 비밀번호, 검색, 에러 상태)
- 카드
- 네비게이션 바 (상단/하단)
- 모달/팝업
- 리스트 아이템
- 탭
- 뱃지/태그
- 토스트/알림

**확정 후 → HTML 프리뷰 생성**

`design-preview/components.html` 파일을 만든다:
- 모든 컴포넌트를 한 페이지에 나열
- 각 컴포넌트의 다양한 상태(hover, active, disabled 등) 표시
- 모바일/데스크톱 양쪽 크기로 미리보기 가능
- 실제 서비스 맥락에 맞는 텍스트 사용

프리뷰를 보여준 뒤 피드백을 받는다.

**확정 즉시 → `design-system.md`에 "컴포넌트 목록" 섹션을 기록한다.**

## Phase 6: 아이콘

기획서의 기능 목록을 분석해서 필요한 아이콘을 정리한다.

**진행 방식:**

1. 기획서 기반으로 필요한 아이콘 목록을 먼저 작성한다
2. 무료 아이콘 라이브러리에서 사용 가능한 것을 매칭한다
   - 추천 라이브러리: Lucide Icons, Heroicons, Phosphor Icons
   - 라이브러리 선택도 사용자와 논의 (스타일 통일이 중요하므로 하나로 통일)
3. 라이브러리에 없는 아이콘은 SVG로 직접 제작할 수 있다고 안내한다
4. 목록을 보여주고 확인받는다

**확정 즉시 → `design-system.md`에 "아이콘" 섹션을 기록한다.**

**대화 예시:**
> 기획서 기반으로 필요한 아이콘 15개를 정리했습니다:
>
> **Lucide Icons에서 사용 가능 (13개):** 홈, 검색, 설정, 알림, 프로필, ...
> **직접 제작 필요 (2개):** 포인트 적립, 커스텀 뱃지
>
> 아이콘 스타일은 Lucide Icons(가는 선 스타일)로 통일하려 합니다. 괜찮으세요?

## Phase 7: MVP 제작

Phase 1~6에서 확정된 모든 디자인 결정을 적용해서 실제 동작하는 MVP를 제작한다.

### 제작 전 확인: 디자인 시스템 문서

Phase 1~6에서 이미 `design-system.md`에 기록되어 있으므로, 빠진 항목이 없는지 최종 확인한다.

```markdown
# 디자인 시스템

## 디자인 컨셉
[Phase 1에서 확정된 컨셉 키워드와 설명]

## 컬러 팔레트
[Phase 2에서 확정된 전체 컬러 목록 — 헥스값, 용도 설명]

## 타이포그래피
[Phase 3에서 확정된 폰트, 크기 체계, 굵기, 줄간격]

## 화면 설정
[Phase 4에서 확정된 브레이크포인트, 레이아웃 전략]

## 컴포넌트 목록
[Phase 5에서 확정된 컴포넌트 이름과 상태]

## 아이콘
[Phase 6에서 확정된 아이콘 목록, 사용 라이브러리, 직접 제작 목록]
```

### 기술 스택 논의

MVP를 어떤 기술로 만들지 사용자와 논의한다. 사용자의 기술 수준에 맞춰 쉬운 말로 설명한다:

- 사용자가 이미 사용하는 기술이 있으면 그것을 우선
- 특별한 선호가 없으면 프로젝트에 적합한 기술을 추천하고 이유를 설명
- 웹 + 모바일 양쪽을 고려한 선택지 제시

### MVP 제작

기술 스택이 확정되면:

1. 프로젝트 구조를 먼저 보여주고 확인받는다
2. 디자인 토큰(컬러, 폰트 등)을 코드에 적용한다
3. Phase 5에서 만든 컴포넌트를 실제 코드로 구현한다
4. 기획서의 핵심 화면들을 구현한다
5. 화면 간 네비게이션을 연결한다
6. 반응형(웹/모바일) 적용한다

제작 중에도 주요 지점마다 "여기까지 확인해보시겠어요?" 하고 체크포인트를 둔다.

## HTML 프리뷰 공통 규칙

Phase 2, 3, 5에서 만드는 HTML 프리뷰 파일의 공통 규칙:

- `design-preview/` 폴더에 저장
- 외부 의존성 없이 단일 HTML 파일로 (CSS 인라인 또는 `<style>` 태그)
- 브라우저에서 바로 열 수 있도록 (서버 불필요)
- 보기 좋게 정리된 레이아웃
- 파일 생성 후 `open` 명령으로 브라우저에서 바로 연다
- 사용자 피드백 후 수정 → 다시 프리뷰 → 확정

## 대화 톤

- 기술 용어는 최소화하고, 사용할 때는 쉽게 풀어서 설명한다
- 선택지를 줄 때는 A/B/C처럼 명확하게 구분한다
- 사용자가 "잘 모르겠어"라고 하면 더 좁은 범위의 질문으로 바꿔서 다시 물어본다
- 한 번에 너무 많은 걸 물어보지 않는다 (한 단계에 1~2개 핵심 질문)
